You are building `ft-nixpalette-river` — a NixOS/Home Manager flake that integrates
the ft-nixpalette theme framework with the River dynamic tiling Wayland compositor.

## What ft-nixpalette provides (the dependency)

ft-nixpalette (github:FT-nixforge/ft-nixpalette) is a theme management flake built on
top of Stylix. At build time it writes two files:

  NixOS:        /etc/ft-nixpalette/colors.json
                /etc/ft-nixpalette/themes.json
  Home Manager: $XDG_DATA_HOME/ft-nixpalette/colors.json
                $XDG_DATA_HOME/ft-nixpalette/themes.json

colors.json structure:
  { "themeId": "builtin:base/gruvbox-dark", "polarity": "dark", "base16": { "base00": "1d2021", ... } }

themes.json structure:
  { "<themeId>": { "themeId": "...", "polarity": "...", "base16": { ... }, "meta": { ... } }, ... }

All base16 hex values are plain 6-character strings WITHOUT a leading #.

## Context: River compositor

River is a minimal dynamic tiling Wayland compositor. It does not have a built-in
wallpaper setter, lock screen, or bar — these are provided by external tools (same
as sway/niri). River is configured via a shell init script (~/.config/river/init).
Home Manager supports river via wayland.windowManager.river.*.

River uses riverctl for runtime configuration — border colors, keybinds etc. can be
applied live via `riverctl border-color-focused` etc.

## What ft-nixpalette-river should provide

A separate flake with nixosModules.default and homeModules.default that
adds River-ecosystem theming on top of ft-nixpalette. All options live under the
`ft-nixpalette-river` namespace.

### 1. swaybg — wallpaper
- Declare `ft-nixpalette-river.swaybg.enable` (default: true)
- Add swaybg exec to wayland.windowManager.river.extraConfig or via exec in init
- Provide `ft-nixpalette-river.swaybg.wallpaper` override option (null or path)

### 2. swaylock — lockscreen
- Declare `ft-nixpalette-river.swaylock.enable` (default: true)
- Same color mapping as ft-nixpalette-sway:
    inside-color → base00, ring-color → base0D, ring-ver-color → base0B,
    ring-wrong-color → base08, text-color → base05, line-color → base03

### 3. River border colors
- Declare `ft-nixpalette-river.borders.enable` (default: true)
- Set via wayland.windowManager.river.extraConfig (riverctl commands):
    riverctl border-color-focused   0x<base0D>
    riverctl border-color-unfocused 0x<base03>
    riverctl border-color-urgent    0x<base08>
- Note: river colors use 0xRRGGBB format (0x prefix, no #)

### 4. waybar
- Declare `ft-nixpalette-river.waybar.enable` (default: false)
- Inject CSS @define-color variables for all 16 base16 slots
- Include note on river-specific waybar module (river/tags)

### 5. mako — notification daemon
- Declare `ft-nixpalette-river.mako.enable` (default: true)
- Same color mapping as ft-nixpalette-sway

### 6. fuzzel / rofi / bemenu launcher
- Declare `ft-nixpalette-river.fuzzel.enable`, `ft-nixpalette-river.bemenu.enable`
- fuzzel: same color mapping as ft-nixpalette-niri
- bemenu: set --tb --tf --fb --ff --nb --nf --hb --hf flags from base16 colors

### 7. Live theme switcher script
- Provide package `ft-nixpalette-river.packages.${system}.ft-nixpalette-switch`
- Script reads themes.json, uses fzf or fuzzel picker, then on selection:
    1. pkill swaybg && swaybg -i <wallpaper> -m fill &
    2. Apply border colors live via riverctl commands (no restart needed!)
    3. pkill -SIGUSR2 waybar
    4. makoctl reload
    5. Writes ~/.cache/ft-nixpalette/active.json
- River is particularly well-suited to live color switching via riverctl

## Architecture requirements

- flake input: inputs."ft-nixpalette".url = "github:FT-nixforge/ft-nixpalette"
- Re-export ft-nixpalette's modules
- Do NOT re-declare ft-nixpalette options
- River border colors can be applied 100% at runtime via riverctl — the switcher
  can change borders without any config file write. Do this for snappier switching.
