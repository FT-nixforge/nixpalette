You are building `ft-nixpalette-hyprland` — a NixOS/Home Manager flake that
integrates the ft-nixpalette theme framework with the Hyprland compositor ecosystem.

## What ft-nixpalette provides (the dependency)

ft-nixpalette (github:FT-nixforge/ft-nixpalette) is a theme management flake built on
top of Stylix. At build time it writes two files:

  NixOS:        /etc/ft-nixpalette/colors.json
                /etc/ft-nixpalette/themes.json
  Home Manager: $XDG_DATA_HOME/ft-nixpalette/colors.json
                $XDG_DATA_HOME/ft-nixpalette/themes.json

colors.json structure:
  { "themeId": "builtin:base/dracula", "polarity": "dark", "base16": { "base00": "282a36", ... } }

themes.json structure:
  { "<themeId>": { "themeId": "...", "polarity": "...", "base16": { ... }, "meta": { ... } }, ... }

The base16 slots follow the standard semantic conventions (base00=background,
base05=foreground, base0D=blue/functions, base0E=purple/keywords, etc.)
All hex values are plain 6-character strings WITHOUT a leading #.

ft-nixpalette also exposes nixosModules.default and homeModules.default which
wire into Stylix for system-wide theming.

## What ft-nixpalette-hyprland should provide

A separate flake with its own nixosModules.default and homeModules.default
that adds Hyprland-ecosystem theming on top of ft-nixpalette. It should:

### 1. swww — animated wallpaper daemon
- Declare a `ft-nixpalette-hyprland.swww.enable` option (default: true)
- On activation / theme switch, call `swww img <wallpaper>` using the path from
  colors.json or a user-provided override
- Support static images and GIFs equally (swww handles both natively)
- Provide a `ft-nixpalette-hyprland.swww.animatedWallpaper` option (null or path)
  that takes priority over the static wallpaper when set

### 2. hyprlock — lockscreen
- Declare a `ft-nixpalette-hyprland.hyprlock.enable` option (default: true)
- Generate a hyprlock.conf that uses base16 colors from colors.json:
    background color → base00
    input field background → base01
    input field border (normal) → base03
    input field border (focused) → base0D
    text color → base05
    error text color → base08
- Use Home Manager's programs.hyprlock.extraConfig or the raw config file approach

### 3. waybar
- Declare a `ft-nixpalette-hyprland.waybar.enable` option (default: false, non-intrusive)
- Inject a CSS variables block into the user's waybar style.css:
    @define-color base00 #<base00>;
    @define-color base01 #<base01>;
    ... all 16 slots ...
- This lets users reference @base0D etc. in their own waybar CSS without
  hardcoding colors

### 4. rofi / wofi theme
- Declare `ft-nixpalette-hyprland.rofi.enable` and `ft-nixpalette-hyprland.wofi.enable`
- Generate a matching color theme from base16 colors

### 5. Live theme switcher script
- Provide a package `ft-nixpalette-hyprland.packages.${system}.ft-nixpalette-switch`
- The script reads /etc/ft-nixpalette/themes.json (or $XDG_DATA_HOME equivalent),
  lists available preloaded themes, and lets the user pick one
- On selection it:
    1. Calls swww to change the wallpaper immediately
    2. Writes the new colors to a user runtime file (~/.cache/ft-nixpalette/active.json)
    3. Reloads waybar (pkill -SIGUSR2 waybar)
    4. Optionally sends a notification (notify-send)
- This gives INSTANT theme switching for colors/wallpaper without any rebuild
  (only works for themes already in themes.json via ft-nixpalette.preloadThemes)

## Architecture requirements

- The flake takes ft-nixpalette as an input and re-exports its modules:
    inputs."ft-nixpalette".url = "github:FT-nixforge/ft-nixpalette";
- ft-nixpalette-hyprland modules should NOT re-declare ft-nixpalette options — they
  read from config."ft-nixpalette".* (which ft-nixpalette declares) and from the JSON files
- All options live under the `ft-nixpalette-hyprland` namespace
- All features are opt-in with `enable` options, defaulting to true only for
  core features (swww, hyprlock) and false for potentially conflicting ones (waybar)
- The Home Manager module handles user-space concerns (hyprlock, waybar CSS,
  rofi/wofi, the switcher script in PATH)
- The NixOS module handles system-space concerns (swww service if run as a
  systemd user service)
