# Task: Design and Implement `nixpalette` — A Reusable NixOS Theme Framework Flake

## Context

You are building `nixpalette`, a standalone, reusable Nix flake that provides a clean theme management framework layered on top of **Stylix**. It introduces folder-based theme definitions, a parent-child inheritance model, and a clear separation between built-in (flake-shipped) and user-defined themes.

This is an **open-source project** — not a personal config snippet. Design it as if strangers will use it.

---

## Hard Requirements

### Flake structure
- Expose `nixosModules.default`
- Expose `homeManagerModules.default`
- Expose `lib` helpers (theme loading, resolution, merging)
- Stylix is a flake input; `nixpalette` wraps it — do not fork or patch it
- Stylix handles application; `nixpalette` handles selection and resolution

### Theme storage layout
```
<themeRoot>/
  base/          # standalone, full themes
  derived/       # themes that declare a parent and override selectively
```

**Defaults:**
- Built-in themes ship inside the flake at `themes/base/` and `themes/derived/`
- User theme root is **configurable**, defaulting to `assets/themes` (relative to the user's flake)

### Each theme is a folder containing:
**Required:**
- `theme.nix` — structured settings file (colors, fonts, wallpaper path, optional metadata)

**Optional:**
- Wallpaper file (referenced in `theme.nix`)
- `meta.nix` — author, description, tags, preview path
- Overrides for specific apps or terminals

### Theme identifiers
Use **namespaced identifiers** to prevent ambiguity:
- `builtin:base/<name>`
- `builtin:derived/<name>`
- `user:base/<name>`
- `user:derived/<name>`

These must be used consistently in inheritance declarations and theme selection.

---

## Inheritance System

### Rules (implement all of these):
1. Derived themes declare exactly one `parent` using the namespaced ID
2. Parent fields are deep-merged with child fields; child wins on collision
3. Allowed inheritance paths:
   - `builtin:derived` -> `builtin:base`      (standard)
   - `user:base` -> nothing (standalone)      (standard)
   - `user:derived` -> `builtin:base`         (primary use case)
   - `user:derived` -> `builtin:derived`      (advanced, must be documented)
   - `user:derived` -> `user:base`            (standard)
4. Circular inheritance is a **hard error** — detect and report it clearly at eval time
5. Deriving from `user:derived` or `builtin:derived` is supported but flagged as advanced
6. No implicit fallback — if a parent ID can't be resolved, fail loudly

## Public Options Interface

Expose a clean NixOS/HM option set. Example shape (you may refine naming):

```nix
nixpalette = {
  enable = true;

  # Namespaced theme ID
  theme = "builtin:base/catppuccin-mocha";

  # Path to user theme root (contains base/ and derived/ subdirs)
  userThemeDir = ./assets/themes;

  # Optionally override stylix settings beyond what the theme provides
  stylixOverrides = { };
};
```

## Architecture Constraints

| Concern                          | Location                  |
|----------------------------------|---------------------------|
| Theme file loading               | `lib/loader.nix`          |
| Inheritance resolution + merging | `lib/resolver.nix`        |
| Stylix integration               | `modules/stylix.nix`      |
| NixOS module (options + wiring)  | `modules/nixos.nix`       |
| HM module (options + wiring)     | `modules/hm.nix`          |
| Built-in themes                  | `themes/base/`, `themes/derived/` |
| Flake entry point                | `flake.nix`               |

Keep these concerns strictly separated — no loading logic in modules, no option declarations in lib.

---

## Future-proofing Requirements

Design explicitly for these **future extensions** (do not implement them now, but do not block them):

1. **Theme switcher** — themes must be discoverable by scanning folder structure; the selected theme ID must be a plain string option that a future CLI/UI can write
2. **Live preview** — `meta.nix` should support a `preview` image path field
3. **Per-app overrides** — `theme.nix` should have an optional `overrides` attrset for named applications
